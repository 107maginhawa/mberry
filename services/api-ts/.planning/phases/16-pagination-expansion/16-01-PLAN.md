# Plan 16-01: Fix TypeSpec pagination emission + unpaginated handlers

## Objective

1. Investigate why OpenAPI generator doesn't emit pagination params from TypeSpec OffsetPaginationParams
2. Add limit/offset to the 5 highest-risk unpaginated list handlers (person, certificates, elections, reviews, storage)

## Tasks

<task id="1">
<title>Investigate TypeSpec→OpenAPI pagination emission</title>
<read_first>
- specs/api/src/common/pagination.tsp
- specs/api/dist/openapi/openapi.json (search for offset or limit params on any endpoint)
- specs/api/src/association/core/engagement.tsp (uses OffsetPaginationParams — check how)
</read_first>
<action>
Check why OffsetPaginationParams doesn't emit to OpenAPI:

1. Read `pagination.tsp` — check if OffsetPaginationParams uses `@query` decorators on each field
2. Check how engagement.tsp spreads `...OffsetPaginationParams` — is it in the operation signature or body?
3. Look at the generated OpenAPI JSON for an endpoint that uses OffsetPaginationParams (like listEngagementActivities from engagement.tsp) — are offset/limit present as query parameters?
4. If NOT present: the issue is likely that `...OffsetPaginationParams` needs to be in the query parameter position, not just spread. Check TypeSpec docs for how to declare query params via model spreading.
5. Run `cd specs/api && bun run build` after any fix to regenerate OpenAPI

If the issue is in pagination.tsp (missing @query decorators), fix it. If it's a TypeSpec compiler limitation, document it and move to task 2.
</action>
<acceptance_criteria>
- Root cause of pagination non-emission is identified
- If fixable: pagination.tsp updated, `bun run build` succeeds, OpenAPI JSON shows offset/limit params on at least one list endpoint
- If not fixable: documented in a comment in pagination.tsp with workaround
</acceptance_criteria>
</task>

<task id="2">
<title>Add pagination to 5 unpaginated list handlers</title>
<read_first>
- services/api-ts/src/handlers/dues/listDuesPayments.ts (reference — already has pagination pattern)
- services/api-ts/src/handlers/person/listPersons.ts
- services/api-ts/src/handlers/certificates/listCertificates.ts
- services/api-ts/src/handlers/elections/listElections.ts
</read_first>
<action>
For each of these 5 handlers, add pagination following the existing pattern from listDuesPayments.ts:

```typescript
const limit = Math.min(parseInt(ctx.req.query('limit') ?? '25', 10), 100);
const offset = parseInt(ctx.req.query('offset') ?? '0', 10);
```

Then pass limit/offset to the repo query:
```typescript
.limit(limit)
.offset(offset)
```

Handlers to update:
1. `services/api-ts/src/handlers/person/listPersons.ts` — if it returns all persons
2. `services/api-ts/src/handlers/certificates/listCertificates.ts` — if it lists all certs
3. `services/api-ts/src/handlers/elections/listElections.ts` — if it lists all elections  
4. `services/api-ts/src/handlers/reviews/listReviews.ts` — if it lists all reviews
5. `services/api-ts/src/handlers/storage/listFiles.ts` — if it lists all files

For each: read the handler first. If it already has pagination (limit/offset), skip it. If the handler delegates to a repo that already supports limit/offset, just pass the query params through.

Cap limit at 100 to prevent unbounded queries.
</action>
<acceptance_criteria>
- Each updated handler parses limit and offset from query params
- Each handler caps limit at 100
- Each handler passes limit/offset to the database query
- No handler returns unbounded result sets
</acceptance_criteria>
</task>

## must_haves
- Root cause of TypeSpec→OpenAPI pagination gap identified
- At least 5 high-volume list endpoints have pagination in handlers
- No unbounded list queries on person, certificates, elections, reviews, storage

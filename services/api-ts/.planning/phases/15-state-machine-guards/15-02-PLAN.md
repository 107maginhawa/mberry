# Plan 15-02: Event and training status transition guards

## Objective

Add status transition guards to event and training update handlers. Currently `updateEvent` and `updateTraining` accept status changes via spread operator with zero validation. Status should only change through dedicated handlers (publishEvent, cancelEvent, etc).

## Tasks

<task id="1">
<title>Strip status from updateEvent spread and add guard</title>
<read_first>
- services/api-ts/src/handlers/events/updateEvent.ts
- services/api-ts/src/handlers/events/cancelEvent.ts
- services/api-ts/src/handlers/association:operations/repos/events.schema.ts
</read_first>
<action>
In `updateEvent.ts`:
1. Destructure `status` out of the body to prevent it from being spread into the update:
```typescript
const { status: _status, ...updateFields } = body;
```
2. If `status` was provided in the body, throw an error:
```typescript
if (body.status !== undefined) {
  throw new BusinessLogicError(
    'Status cannot be changed via update. Use publish, cancel, or complete endpoints.',
    'STATUS_UPDATE_NOT_ALLOWED'
  );
}
```
3. Use `updateFields` instead of `body` in the repo update call.

In `cancelEvent.ts`:
1. Fetch the current event and check status before cancelling:
```typescript
const existing = await repo.findById(id);
if (!existing) throw new NotFoundError('Event not found');
if (existing.status === 'cancelled') {
  throw new BusinessLogicError('Event is already cancelled', 'EVENT_ALREADY_CANCELLED');
}
if (existing.status === 'completed') {
  throw new BusinessLogicError('Cannot cancel a completed event', 'EVENT_COMPLETED');
}
```
</action>
<acceptance_criteria>
- updateEvent.ts contains `STATUS_UPDATE_NOT_ALLOWED` error
- updateEvent.ts destructures status out of body before spreading
- cancelEvent.ts contains guard against cancelling already-cancelled events
- cancelEvent.ts contains guard against cancelling completed events
</acceptance_criteria>
</task>

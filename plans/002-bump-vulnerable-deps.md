## Why this matters

`bun audit` reports three actionable advisories on packages reachable from the
running API and its build:

- **hono** (installed `4.12.22`, manifest `^4.0.0`) — fixed in `>=4.12.25`:
  - `GHSA-88fw-hqm2-52qc` (**HIGH**, CVSS 7.1, CWE-942): CORS middleware reflects
    any Origin with credentials when `origin` defaults to wildcard.
  - `GHSA-wwfh-h76j-fc44` (moderate): path traversal in `serve-static` on Windows.
  - `GHSA-j6c9-x7qj-28xf` (moderate): AWS Lambda Set-Cookie merge.
  - `GHSA-rv63-4mwf-qqc2` (moderate): body-limit bypass on AWS Lambda.
- **form-data** (`>=4.0.0 <4.0.6`) — `GHSA-hmw2-7cc7-3qxx` (**HIGH**, CVSS 7.5,
  CWE-93): CRLF injection via unescaped multipart field/file names. Transitive.
- **@opentelemetry/core** (`<2.8.0`) — `GHSA-8988-4f7v-96qf` (moderate, CWE-770):
  unbounded memory allocation in W3C Baggage propagation (DoS vector).

Reachability note for context (do NOT let this block the bump): Memberry
configures explicit CORS origins (`CORS_ORIGINS` in `services/api-ts/src/core/config.ts`)
rather than the wildcard default, which limits exposure to the hono CORS advisory
— but staying on a patched runtime is the right baseline regardless, and the
form-data CRLF advisory affects multipart uploads (documents/certificates).

After this plan: `bun audit` reports no HIGH/moderate advisories for these three
packages, and the suite still passes.

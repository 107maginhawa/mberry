## Findings considered and rejected (do not re-audit)

- **SDK type drift (`amount: bigint` vs number, nested envelopes, `totalAmount`
  field, `as any` casts)** — documented, deliberate frozen-engine/SDK pattern;
  SDK regen + engine changes are explicitly out of lean scope. Not a defect.
- **"Raw `<div>`/`<h1>` layout = design-law violation"** — false. The design law
  forbids forking *components* (Button/Card/Input), which the app correctly uses
  from `@monobase/ui`. Tailwind layout divs are normal. A Box/Flex/Stack
  primitive set would be over-engineering.
- **Extract `Number()` money coercion into a shared helper** — over-abstraction.
  `Number()` is stdlib; a wrapper adds indirection without value. The only useful
  version (NaN-guarding) is already mitigated: the engine sends valid bigints and
  the pay-link ConfirmDialog renders the actual `₱` amount before any send.
- **PayMongo secret key held in React state** — standard, acceptable:
  `type="password"`, cleared on success, never pre-filled, server never returns
  it (`PaymentSettings.tsx`). Ref-based refactor is marginal.
- **Custom-amount float precision (`parseFloat * 100`)** — mitigated: the
  ConfirmDialog shows `centavosToPhp(centavos)` before send, so the officer sees
  and confirms the exact amount.
- **SendLink custom-amount visible `<label>`** — the input has a correct
  `aria-label` (WCAG AA satisfied); a visible label is polish, not a defect.
- **Cross-org IDOR via the SendLink invoice query key** — overstated by a
  subagent. `membershipId` is globally unique, so no cross-org cache collision is
  possible. Downgraded to a one-line consistency fix (folded into plan 003).

## Newly discovered during execution

- **`officer-flow.spec.ts` is pre-existing red** (found while running plan 004).
  The org app migrated to passwordless email-OTP, but this E2E spec still drives
  a Password field (`getByLabel('Password')`) that no longer exists, so it times
  out. It has been failing since the OTP migration, independent of plans 001-004.
  Plan 004's new specs deliberately model on `import-flow.spec.ts` (localStorage
  org seed) instead. **Plan 005** (TODO) should port `officer-flow.spec.ts` to
  the OTP form or the import-flow auth pattern. Until then the full org E2E suite
  has one known red unrelated to this batch.

## Direction options (not planned — for the maintainer to weigh)

These are product options, not defects. None were turned into plans; raise one
to a plan if you want to pursue it.

- **Batch pay-link send from DuesView** — officers currently send links one
  member at a time; a "send to all overdue" bulk action directly accelerates the
  product's whole wedge (dues collection). Infra exists (`use-bulk-send` already
  bulk-sends from the Roster). Med effort.
- **Tagalog i18n** — `DESIGN.md` flags it; target users are older PH dentists.
  Front-loaded effort; reasonable to defer until a chapter is paying.
- **Roster bulk tier/status mutations** — selection infra exists but only
  send-links is wired; bulk re-tier / mark-standing is the adjacent gap. Med
  effort, needs careful validation UX.

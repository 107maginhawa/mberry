<objective>
Create a membership-lifecycle service that centralizes the payment→membership orchestration currently scattered across multiple handler files. Eliminate duplicated status-check and transaction patterns.
</objective>

<context>
See [15-CONTEXT.md](./15-CONTEXT.md) and [15-PATTERNS.md](./15-PATTERNS.md) for domain remediation background.

**Current problem**: Payment→membership orchestration is duplicated across:
- `settle-payment.ts` — fund allocation + expiry extension
- `markDuesInvoicePaid.ts` — payment status update + settlement call
- `recordDuesPayment.ts` — payment creation + settlement + status update
- `refundDuesPayment.ts` — refund with fund reversal + expiry reversal

Each file reimplements status checks, transaction wrapping, and membership status recomputation. Plans 01-03 fix the immediate bugs but leave the duplication. This plan consolidates into a single service.

**Depends on plans 01, 02, 03**: The transaction fixes and refund logic must land first so this plan can extract already-correct code rather than moving buggy code.
</context>

<tasks>
## Create lifecycle service

1. Read the post-fix versions of:
   - `refundDuesPayment.ts` (plan 01 output)
   - `approveMembershipApplication.ts` (plan 02 output)
   - `recordDuesPayment.ts` (plan 03 output)
   - `settle-payment.ts`
   - `markDuesInvoicePaid.ts`
   - `computeMembershipStatus.ts`
2. Identify shared patterns:
   - Transaction wrapping
   - Fund allocation creation/reversal
   - Expiry date computation/reversal
   - Membership status recomputation
   - Payment status transitions

3. Create `services/api-ts/src/handlers/association:member/utils/membership-lifecycle.ts`:
   ```typescript
   export const membershipLifecycle = {
     settlePayment(txDb, params): Promise<SettlementResult>
     processRefund(txDb, params): Promise<RefundResult>
     recomputeStatus(txDb, membershipId): Promise<MembershipStatus>
   }
   ```
   - `settlePayment`: allocate funds, extend expiry, update membership status
   - `processRefund`: create reversal entries, reset expiry, recompute status
   - `recomputeStatus`: shared status recomputation logic

4. Migrate `settle-payment.ts` to delegate to lifecycle service:
   - Keep the file as a thin wrapper for backward compatibility
   - Core logic moves to `membership-lifecycle.ts`

## Rewire handlers

5. Update `recordDuesPayment.ts` to call `membershipLifecycle.settlePayment()`
6. Update `markDuesInvoicePaid.ts` to call `membershipLifecycle.settlePayment()`
7. Update `refundDuesPayment.ts` to call `membershipLifecycle.processRefund()`

## Verify

8. Run all existing tests for the modified handlers — must pass
9. Run full API test suite to catch any broken imports
10. Verify no duplicated status-transition logic remains across handler files
</tasks>

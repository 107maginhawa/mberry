<objective>
Fix the live refundDuesPayment handler to actually reverse fund allocations, reset duesExpiryDate, and recompute membership status — not just flip payment status to 'refunded'.
</objective>

<context>
See [15-CONTEXT.md](./15-CONTEXT.md) and [15-PATTERNS.md](./15-PATTERNS.md) for domain remediation background.

**Current bug**: `services/api-ts/src/handlers/association:member/refundDuesPayment.ts` only updates the payment row status to 'refunded'. It does NOT:
1. Reverse fund allocation entries created by settlePayment
2. Reset duesExpiryDate on the membership row to the pre-payment value
3. Recompute membership status after the expiry change

**Dead code reference**: `services/api-ts/src/handlers/dues/refundPayment.ts` contains better logic that:
- Fetches original fund allocations for the payment
- Computes refundRatio (refundAmount / originalAmount) for partial refunds
- Creates reversal entries with `isReversal=true` and negative amounts
- This logic was never wired into the live handler
</context>

<tasks>
## RED phase — write failing tests first

1. Read the live `refundDuesPayment.ts` handler and the dead `dues/refundPayment.ts` to understand both implementations
2. Read `settle-payment.ts` to understand how fund allocations and expiry extensions are created
3. Create `refundDuesPayment.test.ts` with these test cases:
   - **Full refund reverses fund allocations**: After refund, reversal entries exist with `isReversal=true` and negative amounts matching originals
   - **Full refund resets duesExpiryDate**: Membership duesExpiryDate reverts to the value before the payment was settled
   - **Partial refund creates proportional reversals**: 50% refund creates entries with 50% of original amounts, negative
   - **Refund recomputes membership status**: After expiry reset, `computeMembershipStatus` is called and membership status updated
   - **Transaction atomicity**: If any step fails, all changes roll back (payment stays 'paid', no reversal entries, expiry unchanged)
   - **Refund of already-refunded payment returns error**
4. Run tests — confirm they fail

## GREEN phase — fix the handler

5. Inline the dead code's fund reversal logic into `refundDuesPayment.ts`:
   - Fetch original fund allocations for the paymentId
   - Compute `refundRatio = refundAmount / payment.amount`
   - Create reversal entries: `amount = -(original.amount * refundRatio)`, `isReversal = true`
6. Add expiry reversal:
   - For full refund: reset `duesExpiryDate` to the value stored before the payment extended it (or subtract the extension period)
   - For partial refund: proportionally reduce the expiry extension
7. Call `computeMembershipStatus()` after expiry change to update membership status
8. Wrap ALL operations in `db.transaction()`:
   - Update payment status to 'refunded' (or 'partially_refunded')
   - Insert reversal fund allocation entries
   - Update membership duesExpiryDate
   - Update membership status
9. Run tests — confirm they pass

## REFACTOR phase

10. Extract shared refund logic if it will be reused by the lifecycle service (plan 04)
11. Ensure no broken imports or unused dead code references
</tasks>

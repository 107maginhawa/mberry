<objective>
Wrap the application approval + membership creation in a single db.transaction() so a failed createOne() doesn't leave the application stuck in 'approved' with no membership record.
</objective>

<context>
See [15-CONTEXT.md](./15-CONTEXT.md) and [15-PATTERNS.md](./15-PATTERNS.md) for domain remediation background.

**Current bug**: `services/api-ts/src/handlers/association:member/approveMembershipApplication.ts` lines 45-54 perform two separate DB writes:
1. Update application status to 'approved'
2. Call `createOne()` to create the membership record

If step 2 fails (e.g., unique constraint violation, DB error), the application is left in 'approved' status with no corresponding membership. The user sees "approved" but has no membership. Re-submitting or re-approving may fail or create duplicates.
</context>

<tasks>

2. Read existing `approveMembershipApplication.test.ts` to understand current test coverage

   - **Transaction rollback on membership failure**: Mock `createOne()` to throw, verify application status is NOT changed to 'approved'
   - **Happy path still works**: Application approved + membership created in single transaction
   - **Transaction uses txDb**: Verify both operations receive the transaction-scoped db instance
4. Run tests — confirm new tests fail (rollback test will fail because current code doesn't use transaction)

7. Run tests — confirm all pass (including existing tests)

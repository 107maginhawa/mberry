# Plan 16-01: Transfer Lifecycle E2E Tests

## Goal
Validate the full affiliation transfer workflow end-to-end: member initiates → source officer approves → target officer approves → transfer completes. Also test deny and invalid-state transitions.

## Context
- 7 transfer handlers exist under `services/api-ts/src/handlers/association:member/`
- Existing E2E tests (`apps/memberry/tests/e2e/member/transfer.spec.ts`) only test dialog UI
- Transfer state machine: requested → pendingSourceApproval → pendingTargetApproval → approved → completed
- API endpoints:
  - `POST /association/member/affiliation-transfers` (create)
  - `POST /association/member/affiliation-transfers/{id}/approve-source`
  - `POST /association/member/affiliation-transfers/{id}/approve-target`
  - `POST /association/member/affiliation-transfers/{id}/complete`
  - `POST /association/member/affiliation-transfers/{id}/deny`
  - `GET /association/member/affiliation-transfers/{id}`
  - `GET /association/member/affiliation-transfers`

## Approach
Write API-level integration tests (Bun test) for the transfer lifecycle since the full flow requires multiple actors (member, source officer, target officer) which E2E browser tests can't easily orchestrate.

## Tasks

### Task 1: Create transfer lifecycle test file
- File: `services/api-ts/src/handlers/association:member/transfer-lifecycle.test.ts`
- Use existing test patterns from the `association:member` directory
- Import `apiAs` helper or use direct API calls with test auth

### Task 2: RED — Happy path lifecycle test
```
Test: "transfer lifecycle: create → source approve → target approve → complete"
1. Member creates transfer (POST /affiliation-transfers) → 201, status=requested
2. Source officer approves (POST /{id}/approve-source) → 200, status=pendingTargetApproval
3. Target officer approves (POST /{id}/approve-target) → 200, status=approved
4. Complete transfer (POST /{id}/complete) → 200, status=completed
5. GET /{id} — verify final state has completedAt set
```

### Task 3: RED — Deny flow test
```
Test: "transfer can be denied at any pre-completion state"
1. Create transfer → status=requested
2. Deny transfer (POST /{id}/deny) → 200, status=denied
3. Verify cannot approve denied transfer (should throw BusinessLogicError)
```

### Task 4: RED — Invalid state transition tests
```
Test: "cannot complete unapproved transfer"
1. Create transfer → status=requested
2. POST /{id}/complete → error (status must be 'approved')

Test: "cannot approve completed transfer"
1. Run full lifecycle to 'completed'
2. POST /{id}/approve-source → error

Test: "cannot deny completed transfer"
1. Run full lifecycle to 'completed'
2. POST /{id}/deny → error
```

### Task 5: RED — Reverse approval order test
```
Test: "target approves before source → still reaches approved state"
1. Create transfer → requested
2. Target approves first → pendingSourceApproval
3. Source approves → approved (both approvals present)
4. Complete → completed
```

### Task 6: GREEN — Fix any handler bugs discovered by tests

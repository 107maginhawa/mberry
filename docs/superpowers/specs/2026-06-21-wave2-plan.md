## 2. Confirmed cluster split + rationale

The proposed 4-cluster split is **CONFIRMED**. Each cluster is justified by a **shared real-PG harness or domain-event test rig** (the dominant Wave-2 cost), not just thematic grouping. Verified against source (`pg-scratch.ts` exists; `events.schema.ts` is shared by eventsâ†”association:operations; surveys/reviews couple to association:member; storage/comms/platformadmin already ship genuine real-PG integration tests; elections has none and member/governance does).

### W2-B1 â€” scheduling: `events`, `booking`, `association:operations` (order 1)
Tightest coupling in the wave. events imports tables + `EventsRepository` chains directly from `association:operations/repos/events.schema.ts`; the w2a paid-registration flow reuses the same `event` / `event_registration` / `check_in` tables. **One shared `createScratch` harness** (new `src/test-utils/scheduling-fixtures.ts` seed helper) serves all three. Highest leverage: a single harness slice unblocks the P0 `registerAtomic` FOR-UPDATE concurrency proof, the migration-0078 `org_id NOT NULL` invariant, and the cross-module Stripe settlement seam.

### W2-B2 â€” content: `documents`, `surveys`, `reviews` (order 2)
Three org-scoped CRUD modules with **no DB FKs into scheduling**. All three need a person/org/membership seed (surveys `utils/membership-check.ts` + reviews `person` restrict-FK both couple to `association:member`; documents is zero-FK but org-scoped), so **one content scratch fixture** serves the trio. None has any real-PG today (documents = makeDb stub, surveys = fakeDb chain, reviews = stale factory). Each carries an untested DB-constraint surface (jsonb containment + 0078 org_id NOT NULL; partial-unique dedup; 4 CHECK/unique + restrict-FK cascade ordering). No mis-clustering: documents does **not** couple to scheduling repos despite both being "content-ish".

### W2-B3 â€” feedback: `notifs`, `email` (order 3)
The fan-out / delivery sink. Both org-scoped, low-FK, single-table-per-repo. Both already have a real-PG file that is **CI-gated or partial** (notifs `notification.repo.integration.test.ts` early-returns on CI; email `suppression.repo` real but `queue`/`template` mock-only). **One feedback `createScratch` rig** co-locating `notification` + `email_queue` DDL, plus the shared domain-event fan-out contract (consumers fire-and-forget into both tables).

### W2-B4 â€” realtime-admin: `storage`, `comms`, `platformadmin`, `elections` (order 4)
A **maintenance cluster keyed by remediation TYPE**, not data sharing â€” these modules are self-contained schemas with near-zero mutual coupling. `storage-repos` / `comms-repos` / `platformadmin-repos` integration tests are **all genuine real-PG already**; the work is to run them in a DB-backed CI lane and extend/close specific holes. Two flagged exceptions:
- **storage** is a non-sharer (one tiny single-table repo, already owns its harness). It serves as the *pattern template* siblings copy, not a shared fixture.
- **elections** is the odd member â€” ZERO integration tests, needs a **NEW** ElectionsRepository real-PG harness. It should **REUSE `member/governance`'s existing real-PG fixtures** (`election-lifecycle.integration.test.ts`, `election-officer-term-cascade.integration.test.ts`) which already seed memberships/officer_terms/elections (the exact fixtures BR-34 eligibility + vote-tally need), **not** the realtime-admin DDL. This is flagged explicitly so elections is not mis-built against storage/comms DDL. (We keep elections in B4 rather than re-clustering into association governance because its handlers and Wave-2 ownership sit here; only its *harness source* is borrowed.)

**No re-clustering required.** The only adjustment is the elections harness-source annotation above.

## 3. Cluster ordering (risk Ă— leverage)

1. **W2-B1 scheduling â€” FIRST.** Highest leverage and highest risk: the P0 `registerAtomic` capacity/lock race is currently proven only against scripted mock counts (double-booking past capacity is unverified at SQL); the shared scheduling harness unblocks 3 modules at once and is the cluster other plans depend on. 3. **W2-B3 feedback.** Coverage largely *exists* but is CI-gated; un-gating + the fan-out contract is medium effort, medium risk (silent notification/email drops). Lower than B2 because the SQL is already authored and locally green.
4. **W2-B4 realtime-admin â€” LAST.** Mostly un-gate/extend of harnesses that already pass locally â€” lowest *absence* risk. elections (the one new harness here) is sequenced inside this cluster to ride member/governance fixtures already standing by then.

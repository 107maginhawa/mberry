# Memberry Documentation Landscape

Three overlapping doc sets coexist in this repo by design. This file tells you which is canonical for what so you spend time in the right one.

## At a glance

| Path | Canon for | Format |
|---|---|---|
| [`docs/product/`](./product/) | Product specs — domain model, PRDs, threat model, role/permission matrix, per-module specs (m01-m22) with abstract UI prototypes | OLI-generated PRD set, structured |
| [`docs/ver-3/`](./ver-3/) | Design system + business rules + concrete UX-by-role screen specs | Hand-authored iteration ("version 3") |
| [`docs/execution/`](./execution/) | Active slice plans — VERTICAL_SLICE_PLAN + WAVE0B/1/2A/2B/3A + per-slice SLICE_SPEC.md | Execution tracking, mutates with sprints |
| [`docs/architecture/`](./architecture/) | One-off architecture decisions outside the product PRD set | Hand-authored |
| [`docs/roadmap/`](./roadmap/) | Long-term roadmap notes | Hand-authored |
| [`docs/project-map/`](./project-map/) | Generated code-vs-spec maps (routes, journeys, BR coverage, gaps) | Auto-generated by `generate.ts` |
| [`docs/_archive/`](./_archive/) | Historical OLI process artifacts (audits, compliance, enforcement, trace reports, abandoned spec formats) | Frozen; don't extend |
| [`ARCHITECTURE.md`](./ARCHITECTURE.md) | Top-level technical architecture | Hand-authored |
| [`MULTI-TENANT-AUDIT.md`](./MULTI-TENANT-AUDIT.md) | Multi-tenancy review | Hand-authored |
| [`QA-COVERAGE-MATRIX.md`](./QA-COVERAGE-MATRIX.md) | QA test coverage by feature | Hand-authored |

## How they overlap

### Per-module specs

- [`docs/product/modules/m05-membership/`](./product/modules/m05-membership/) holds the **API contract** (`API_CONTRACTS.md`), **module spec** (`MODULE_SPEC.md`), **navigation map** (`NAVIGATION_MAP.md`), plus an **abstract UI prototype** in `ui-prototype/` (`screens.md`, `components.md`, `interaction-states.md`, `mock-data.md`).
- [`docs/ver-3/ux/screens/member/`](./ver-3/ux/screens/) holds the **concrete per-screen specs** for the member surface (one file per screen — `certificate-detail.md`, `credits-log.md`, `dashboard.md`, etc.).

**Both are canonical.** Product specs describe the module's intent + API; ver-3 screens describe the user-visible surface. If they conflict on a UI detail, **ver-3 wins for the screen**; product wins for the API and module-level behavior.

### Business rules

- [`docs/product/`](./product/) carries `DOMAIN_MODEL.md`, `STATE_MACHINES.md`, `ERROR_TAXONOMY.md`, `EVENT_CONTRACTS.md`, `WORKFLOW_MAP.md`.
- [`docs/ver-3/business/`](./ver-3/business/) carries `business-rules.md`, `cross-cutting.md`, `metrics.md`, `personas-and-roles.md`, `br-registry.json`, `roadmap.md`.

Treat ver-3 business as the **runtime business-rules source of truth** (the `br-registry.json` is what `bun run test:br` consumes). Product specs are referenced for the formal contract.

## When you create a new doc

| You want to document | Put it here |
|---|---|
| A new module's spec, API contract, abstract UI | `docs/product/modules/m{NN}-{slug}/` |
| A concrete screen the user sees | `docs/ver-3/ux/screens/{role}/{screen-slug}.md` |
| A business rule with an ID | Add to `docs/ver-3/business/br-registry.json` |
| A multi-module execution plan | `docs/execution/slices/{slice-slug}/SLICE_SPEC.md` |
| A one-off architectural decision | `docs/architecture/{decision-slug}.md` |
| Anything else | Ask before creating a new top-level subdirectory |

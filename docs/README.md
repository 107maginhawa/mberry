# Memberry Documentation Landscape

Two overlapping doc sets coexist in this repo by design. This file tells you which is canonical for what so you spend time in the right one.

## How they overlap

### Per-module specs

- [`docs/product/modules/m05-membership/`](./product/modules/m05-membership/) holds the **API contract** (`API_CONTRACTS.md`), **product-module spec** (`MODULE_SPEC.md`), **navigation map** (`NAVIGATION_MAP.md`).
- [`docs/product/MODULE_SPEC.member.membership.md`](./product/MODULE_SPEC.member.membership.md) is a **separate, handler-level** spec — written from source inspection, granular per handler directory. Both are canonical for their own layer (product vs handler). See [PRD_INDEX](./product/prd/PRD_INDEX.md) for the full map.
- [`docs/ver-3/ux/screens/member/`](./ver-3/ux/screens/) holds the **concrete per-screen specs** for the member surface (one file per screen — `certificate-detail.md`, `credits-log.md`, `dashboard.md`, etc.).

**All canonical at their own layer.** Product specs describe the module's intent + API; handler specs describe the actual implementation surface; ver-3 screens describe the user-visible surface. Conflict resolution: ver-3 wins for the screen; product wins for the API and module-level behavior; handler spec wins for handler implementation detail.

### Business rules

- [`docs/product/`](./product/) carries `DOMAIN_MODEL.md`, `STATE_MACHINES.md`, `ERROR_TAXONOMY.md`, `EVENT_CONTRACTS.md`, `WORKFLOW_MAP.md`.
- [`docs/ver-3/business/`](./ver-3/business/) carries `business-rules.md`, `cross-cutting.md`, `metrics.md`, `personas-and-roles.md`, `br-registry.json`, `roadmap.md`.

Treat ver-3 business as the **runtime business-rules source of truth** (the `br-registry.json` is what `bun run test:br` consumes). Product specs are referenced for the formal contract.

## When you create a new doc

| You want to document | Put it here |
|---|---|
| A new module's product spec, API contract, abstract UI | `docs/product/modules/m{NN}-{slug}/` |
| A new handler-level spec (source-inspected) | `docs/product/MODULE_SPEC.{handler-name}.md` |
| A concrete screen the user sees | `docs/ver-3/ux/screens/{role}/{screen-slug}.md` |
| A business rule with an ID | Add to `docs/ver-3/business/br-registry.json` |
| A multi-module execution plan | `docs/execution/slices/{slice-slug}/SLICE_SPEC.md` |
| An architecture decision record | `docs/architecture/adr/{NNNN}-{decision-slug}.md` |
| A one-off architectural note | `docs/architecture/{decision-slug}.md` |
| An audit output | `docs/audits/` or `docs/quality/` (per audit type) |
| Anything else | Ask before creating a new top-level subdirectory |

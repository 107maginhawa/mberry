# ADR-0003: Drizzle ORM over Prisma

- Status: Accepted
- Date: 2026-06-06
- Deciders: Memberry team

## Context

The API service needs a database access layer for PostgreSQL. Two dominant TypeScript ORMs were available at the time the Monobase template was established: Prisma and Drizzle.

**Rationale: TBD** — the original pre-template selection rationale was not documented. The following reflects observable properties of the current implementation and the constraints that would have driven the decision.

Observable properties of the Drizzle adoption:
- Schema is defined in TypeScript files colocated with handler directories (`handlers/{module}/repos/*.schema.ts`), not in a separate `schema.prisma` file. This keeps schema ownership close to the module that owns the data.
- Queries use the Drizzle query builder (`db.select().from(...).where(...)`) and prepared statements; no magic string queries.
- Migrations are generated by `drizzle-kit` and land as SQL files in `src/generated/migrations/` — reviewable plain SQL, not an opaque binary diff.
- Drizzle has zero runtime overhead for type inference — types are inferred from the schema at compile time, not via a separate client generation step.
- Drizzle runs natively in Bun (no Node.js adapter required). Prisma required a Rust engine binary and historically had Bun compatibility issues.

Source: `CLAUDE.md` §"Drizzle ORM Usage", `CONTRIBUTING.md` §"Database Workflow", `CONTRIBUTING.md` §"Database Queries".

## Decision

Drizzle ORM is the database access layer for all handler repositories. Prisma is not used and should not be introduced. Schema files live colocated with handlers. Migrations are generated by `drizzle-kit` and are reviewed as plain SQL before merging.

## Consequences

### Positive
- Native Bun compatibility — no Rust engine binary, no platform-specific build step.
- Schema-as-TypeScript enables the module ownership pattern (schema file lives next to the handler that owns the table).
- Plain SQL migration files are human-readable and auditable.
- Zero-overhead type inference from schema at compile time.
- Prepared statements are idiomatic, reducing SQL injection risk.

### Negative / tradeoffs
- Drizzle's query API is more verbose than Prisma's `.findMany()` / `.include()` — relational queries require explicit joins.
- No built-in soft-delete or lifecycle hooks — those are implemented manually in repositories.
- `drizzle-kit` generates migrations but does not run them automatically in CI; the server runs migrations on startup.

### Neutral
- Drizzle Studio (`bun run db:studio`) provides a GUI for local inspection.
- Teams familiar with Prisma face a learning curve with Drizzle's builder API.

## Alternatives considered

- **Prisma** — rejected (inferred) due to Bun compatibility concerns at time of adoption and the need for a separate Rust engine binary that complicates the Docker image and Bun runtime.
- **Raw SQL / `pg` driver** — rejected because type safety from schema to query result would require manual maintenance.
- **Kysely** — not evaluated; Drizzle was already selected in the Monobase template.
